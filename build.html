On this page you'll create a new project in Cursor and use a carefully designed build prompt to generate a working Next.js + InstantDB version of your KCG Ventures ERP.

1. Create your project folder

First, you'll create an empty folder on your computer where all of the app's files will live.

  1. Create a new folder on your computer. Name it something like kcg-ventures-erp (no spaces – use dashes).
  2. Open Cursor.

You don't need to modify this prompt beyond pasting your InstantDB App ID into the two marked places.

Build me a Next.js web application. Here are the complete specifications:

## Tech Stack (DO NOT DEVIATE)
- Next.js 14+ with App Router (not Pages Router)
- TypeScript
- Tailwind CSS for styling
- InstantDB for database and authentication (use @instantdb/react)

## InstantDB Configuration
My InstantDB App ID is: YOUR_APP_ID_HERE

Initialize InstantDB in a file at `src/lib/db.ts` like this:
```typescript
import { init } from '@instantdb/react';
import schema from '../../instant.schema';

const APP_ID = 'YOUR_APP_ID_HERE';

export const db = init({
  appId: APP_ID,
  schema,
});
```

## App Description
Build an app called **KCG Ventures ERP**. It is a focused production planning and inventory management system for a shared production facility. The app replaces complex Google Sheets with a structured web application that models freeze-dryer capacity, yield logging, batch planning based on target dried ingredient amounts, and inventory ledgers with append-only transactions. All key operational variables are configurable through the UI (no hard-coded calibration assumptions).

## Data Model
Create an InstantDB schema in `instant.schema.ts` with these entities and fields:

1. `calibration` (a single global calibration profile for now)
- `id`: string (InstantDB id)
- `number_of_freeze_dryers`: number
- `trays_per_machine_per_cycle`: number
- `cycle_time_hours`: number               // active freeze-drying time per cycle
- `defrost_cleaning_hours`: number         // defrost / cleaning time per cycle
- `operating_hours_per_day`: number
- `operating_days_per_week`: number
- `fresh_load_per_tray_g`: number          // grams of fresh material per tray
- `dry_matter_fraction`: number            // fraction of dry matter in fresh material (0-1)
- `owner_id`: string (user id reference, optional – used to scope per user)

2. `yield_entries`
- `id`: string
- `microgreen_type`: string
- `tray_harvest_date`: date
- `fresh_harvested_weight_g`: number
- `dried_output_weight_g`: number | null
- `tray_identifier`: string | null
- `created_at`: date

3. `batches`
- `id`: string
- `production_start_date`: date
- `production_end_date`: date
- `target_ingredient`: string         // e.g. "broccoli powder"
- `target_dried_weight_g`: number
- `status`: string                    // e.g. "planned", "draft"
- `created_at`: date

4. `inventory_items`
- `id`: string
- `name`: string                      // e.g. "Broccoli floret", "Pouch 250g"
- `unit`: string                      // e.g. "g", "kg", "units"
- `quantity_on_hand`: number         // derived from transactions but kept for fast display
- `created_at`: date

5. `inventory_transactions`
- `id`: string
- `type`: string                      // "purchase", "consumption", "adjustment"
- `quantity`: number
- `reference`: string | null          // optional free-text note
- `created_at`: date

6. `finished_goods_batches`
- `id`: string
- `product_name`: string
- `quantity_produced`: number
- `date_produced`: date
- `expiration_date`: date | null
- `created_at`: date

Relationships:
- Link all user-owned entities to the built-in `$users` table:
  - Each `calibration` has one `owner` (a `$users` record).
  - Each `yield_entries` record belongs to one user.
  - Each `batches` record belongs to one user.
  - Each `inventory_items` record belongs to one user.
  - Each `inventory_transactions` record belongs to one user and one `inventory_items` record.
  - Each `finished_goods_batches` record belongs to one user.

Model these relations using InstantDB links (for example using `links` with forward/reverse definitions).

## Authentication
Implement user authentication using InstantDB's Magic Code flow:
1. User enters their email.
2. InstantDB sends a 6-digit code to that email.
3. User enters the code to log in.
4. Use `db.useAuth()` to check login status.
5. Protect all routes that read or write production data so only authenticated users can access them.

When a user is not logged in:
- Redirect them to the `/login` page when they try to access protected pages.
- Show a simple, friendly message explaining that they need to log in to use the ERP.

  - `per_cycle_fresh_capacity = number_of_freeze_dryers * trays_per_machine_per_cycle * fresh_load_per_tray_g`

    - `total_available_hours = production_days * operating_hours_per_day`
    - `max_cycles_available = floor(total_available_hours / hours_per_cycle)`

4. `/yield`
- A page for logging tray-level yield data and viewing recent history.
- Features:
  - A form to create a new `yield_entries` record with:
    - microgreen_type
    - tray_harvest_date
    - fresh_harvested_weight_g
    - dried_output_weight_g (optional)
    - tray_identifier (optional)
  - A table or list of recent entries (e.g. last 20), sorted by `tray_harvest_date` descending.
  - Calculated rolling averages displayed at the top:
    - Average fresh yield per tray (mean of `fresh_harvested_weight_g`).
    - Average dried yield per tray (mean of `dried_output_weight_g` for entries where it is not null).

5. `/planner`
- A batch planner that starts from a target dried ingredient amount and tests freeze-dryer feasibility.
- Inputs in a form:
  - production_start_date
  - production_end_date
  - target_ingredient (string)
  - target_dried_weight_g (number)
- Use the calibration record and formulas below to calculate feasibility.

### Freeze Dryer Feasibility Model (Important)

Use these formulas exactly, based on the production idea document:

1. Compute the inclusive number of production days:
```ts
const production_days = (endDate - startDate) + 1;
```
Where `startDate` and `endDate` are dates (you can compute this using JavaScript date math and dividing the difference in milliseconds by the length of a day). Days are inclusive, so if start and end are the same day, `production_days` should be 1.

2. Derived from calibration:
```ts
const hours_per_cycle = cycle_time_hours + defrost_cleaning_hours;

const per_cycle_fresh_capacity =
  number_of_freeze_dryers *
  trays_per_machine_per_cycle *
  fresh_load_per_tray_g;

const total_available_hours = production_days * operating_hours_per_day;

const max_cycles_available = Math.floor(total_available_hours / hours_per_cycle);
```

3. Use yield assumptions to go from target dried to required fresh:
- The `dry_matter_fraction` is the fraction of dry matter in the fresh load.
- Given a target dried weight (`target_dried_weight_g`), approximate the fresh weight needed as:
```ts
const fresh_needed_g = target_dried_weight_g / dry_matter_fraction;
```

4. Convert fresh needed into required freeze-dryer cycles:
```ts
const cycles_needed = fresh_needed_g / per_cycle_fresh_capacity;
```

5. Feasibility condition:
- The batch is feasible from a freeze-dryer perspective if:
```ts
const feasible = max_cycles_available >= Math.ceil(cycles_needed);
```

On the `/planner` page, display:
- Trays required (you can approximate by `cycles_needed * trays_per_machine_per_cycle`, rounded up).
- Cycles required (`Math.ceil(cycles_needed)`).
- Total hours required (`Math.ceil(cycles_needed) * hours_per_cycle`).
- Available cycles (`max_cycles_available`).
- A clear status:
  - If feasible: show a success message.
  - If not feasible: show how many cycles, hours, or fresh capacity you are short by.

6. Optional (if scope allows): inventory sufficiency check
- For now, you can implement a simple check where the user manually enters the amount of a key inventory item required, and compare it to `quantity_on_hand` for that item:
```ts
const sufficient = required_quantity <= quantity_on_hand;
```
- This can be presented on the planner page as a second “gate” after freeze-dryer capacity.

6. `/inventory`
- A page that acts as a simple inventory ledger.
- Features:
  - List all `inventory_items` with `name`, `unit`, and `quantity_on_hand`.
  - A form to create new `inventory_items`.
  - A section to record new `inventory_transactions`:
    - Choose an `inventory_item`.
    - Select `type` ("purchase", "consumption", "adjustment").
    - Enter `quantity`.
    - Optional `reference` note.
  - When a transaction is created, also update the `quantity_on_hand` for that item in a consistent way:
    - "purchase": add the quantity.
    - "consumption": subtract the quantity.
    - "adjustment": add the quantity (it can be negative for corrections).
  - Show a history list of recent transactions, sorted by `created_at` descending.

7. `/finished-goods`
- A simple page to log finished goods batches.
- Features:
  - A form to create a `finished_goods_batches` record with:
    - product_name
    - quantity_produced
    - date_produced
    - expiration_date (optional, can be derived using simple rules if you want)
  - A list or table of finished goods batches.

All of the above pages (except `/login`) must be protected so only authenticated users can access them.

In `instant.perms.ts`, configure permissions so that:
- Users can only read and write records that belong to them (based on the links to `$users`).
- Authentication and Magic Code login work as expected.
- It is safe to use this app on the public internet without exposing other users' data.

When you run the build prompt, Cursor will start generating files and folders in your project. You'll see them appear in the sidebar in real time.

  • It may take several minutes – especially when creating multiple pages and the InstantDB schema.

4. Run your app locally

After Cursor finishes generating code, you're ready to see KCG Ventures ERP running on your own machine.

  1. In Cursor, open the integrated terminal (View → Terminal).
  2. Make sure the terminal is pointed at your project folder (it usually is, by default).
  3. Run the development server:
npm run dev

Wait until you see a line that includes something like http://localhost:3000.

What is localhost:3000?

localhost is a special address that means “this computer.” The :3000 part is like a specific doorway, or port, where your app is listening for requests.

When you type http://localhost:3000 into your browser, you're asking your computer: “Show me the web app that is currently running on port 3000.”

Open your browser and go to http://localhost:3000. You should see the homepage of your new KCG Ventures ERP app.

5. Fix issues with the AI's help

It's common for the first version of a generated app to have a few rough edges: TypeScript errors, missing imports, or other small issues. You don't need to debug these manually.

  1. If the terminal shows an error, select the error message text and copy it.
  2. If the browser shows an error screen, copy the error message from there instead.
  3. Open Cursor's Composer (Cmd+I / Ctrl+I).
  4. Paste the error and say something like: "I'm getting this error when I run `npm run dev`. Please fix it for me in this project."
  5. Let Cursor propose code changes. Review them briefly, then accept/apply the edits.
  6. Re-run npm run dev and check again.

This error → ask AI → apply fix loop is normal. Over time you'll understand more of what the AI is doing, but you can start using the app long before you understand every line of code.
